# 抖音 / TikTok 个人归档系统 — 产品需求文档 (PRD)

> 版本：v0.1（初稿，待评审）
> 作者：基于 都喜欢 DoLike 仓库现状整理
> 适用范围：抖音网页端 (douyin.com) 个人账户内容的本地归档与离线浏览

---

## 1. 项目背景与目标

抖音网页端的「我的」页面里散落着：个人作品、喜欢、收藏、稍后再看、收藏夹（视频/音乐/合集）等内容。这些数据完全依赖云端账号，存在**号封即丢、视频下架、收藏夹失效、网络墙阻断回看**等风险。

本项目目标：

1. **抓取并保存**当前登录抖音账号下所有「我相关」的内容（视频文件 + 元数据 + 封面 + 音频）到本地磁盘。
2. **离线浏览**：通过本地 Web 应用复现抖音「我的」页面体验，按发布时间倒序展示、可搜索、可按时长筛选长 / 短视频。
3. **本地账号体系**：本系统并非"抖音第二客户端"，因此需要先用本地账号登录才能使用归档功能（区别于抖音账号）。

**非目标**（v1 明确不做）：

- 不做发布、互动（点赞 / 评论 / 关注）等写操作。
- 不做多用户共享、不做云同步、不做反审查二次分发。
- 不做抖音直播录制、不做实时推流。
- 不做去水印再上传（本地保留原视频文件即可）。

---

## 2. 角色与使用场景

| 角色 | 描述 | 主要诉求 |
| --- | --- | --- |
| **本地用户** | 使用这台机器的抖音重度用户（创作者 / 收藏控） | 备份自己沉淀多年的喜欢与收藏夹；防止内容被平台清理 |
| **多账号用户** | 拥有多个抖音账号（小号 / 工作号） | 在同一本地账号下切换抖音 cookie，归档不同账号的资产 |

典型场景：

- **S1 首次接入**：用户启动应用 → 注册本地账号（用户名 + 密码）→ 登录后看到"未关联抖音账号"提示 → 导入抖音 cookie → 触发全量归档。
- **S2 增量归档**：用户每月手动触发一次"增量同步"，系统对比抖音云端列表与本地数据库，仅下载新增 / 变更项。
- **S3 离线浏览**：抖音原视频被删除后，用户在本地搜索关键词、按"长视频"筛选，仍可流畅播放。
- **S4 内容迁移**：用户换机器，把整个数据目录拷贝到新机继续使用。

## 3. 功能需求

### 3.1 本地账号系统（鉴权）

| ID | 需求 | 说明 |
| --- | --- | --- |
| F1.1 | 首次启动注册 | 数据库无任何账号时，应用进入"注册引导"页，必须创建一个本地用户名 + 密码。 |
| F1.2 | 登录页 | 后续启动进入登录页，输入用户名 + 密码后进入主界面。 |
| F1.3 | 密码存储 | 密码用 bcrypt / argon2 加盐哈希保存于本地 SQLite，不存明文。 |
| F1.4 | 会话保持 | 登录后写入本地 session token（HttpOnly cookie），默认 7 天有效。 |
| F1.5 | 注销 / 修改密码 | 提供基本的修改密码、退出登录入口。 |
| F1.6 | 单用户即可 | v1 默认单用户系统；数据库结构支持多用户但不暴露 UI。 |

**说明**：UI 已存在 `user-modal` 等组件，但缺独立登录 / 注册页，需要新增。

### 3.2 抖音账号接入（三路登录，并存）

抖音网页端登录依赖 `sessionid` 等 cookie + 浏览器指纹。v1 同时实现 3 种方式，三者**互不替代、可并存**，UI 中默认展示顺序：**A → B → C**。任一方式成功都能向后端注入有效的抖音身份，后端统一抽象为 `DouyinSession`。

#### 方案 A：CloakBrowser 无头扫码（**首选**，UI 默认展示）

| ID | 需求 | 说明 |
| --- | --- | --- |
| F2.A.1 | 启动隐身内核 | 后端启动一个 CloakBrowser（或具备指纹伪装能力的 Chromium：Playwright + stealth），打开 `https://www.douyin.com/`。 |
| F2.A.2 | 截图二维码 | 触发抖音登录弹窗 → 截取二维码区域 → 转 base64 → 通过 WebSocket 推到前端展示。 |
| F2.A.3 | 轮询登录态 | 后端轮询页面 cookie，检测到 `sessionid` 后写入 `DouyinSession`，关闭无头窗口。 |
| F2.A.4 | 超时与刷新 | 60s 内未扫码 → 刷新二维码；3 次刷新失败 → 提示用户切换方案 B。 |
| F2.A.5 | 自动降级 | 启动 CloakBrowser 失败（依赖缺失 / 风控阻断）→ 前端自动滑到方案 B。 |

#### 方案 B：手动 Cookie 粘贴（**兜底**）

| ID | 需求 | 说明 |
| --- | --- | --- |
| F2.B.1 | Cookie 导入界面 | 提供文本框，用户从浏览器 DevTools 复制完整 cookie 字符串后粘贴。 |
| F2.B.2 | Cookie 校验 | 后端用该 cookie 调用 `/aweme/v1/web/user/profile/self/`，能拿到 nickname / sec_uid 即视为有效。 |
| F2.B.3 | Cookie 失效提示 | 后续抓取若返回登录态失效，前端 banner 提示重新导入。 |

#### 方案 C：浏览器插件桥接（**并行通道**，可与 A/B 同时使用）

| ID | 需求 | 说明 |
| --- | --- | --- |
| F2.C.1 | 插件功能 | 一个 Chrome / Edge MV3 插件，安装后在 `douyin.com` 页面上注入"一键归档"按钮。 |
| F2.C.2 | 数据采集 | 用户点击后，插件在当前真实登录态下读取「我的收藏 / 喜欢 / 作品 / 合集 / 音乐 / 稍后再看」列表 JSON。 |
| F2.C.3 | 推送本地 | 插件将列表 POST 到 `http://127.0.0.1:7777/api/bridge/push`，由本项目接收并入库。 |
| F2.C.4 | 鉴权 | 推送时携带本地账号生成的一次性 push_token（在 UI 设置页生成 + 复制到插件配置）。 |
| F2.C.5 | 与 A/B 并存 | A 已成功登录时，C 仍可手动触发用于增量推送；A 失效时 C 不受影响；C 可单独承担"列表发现"环节，下载仍走后端。 |
| F2.C.6 | 仅推送列表 | 插件不下载视频本体，下载始终由后端用 A/B 的 cookie 完成；如二者都无效，下载失败入待重试队列。 |

#### 统一身份层 `DouyinSession`

| 字段 | 来源 |
| --- | --- |
| `sec_uid` / `nickname` | A/B 校验时拉取；C 推送时携带 |
| `cookie` | A 截取浏览器 cookie / B 用户粘贴；C 不提供 |
| `source` | `'cloak' | 'manual' | 'bridge'` |
| `is_valid` | 最近一次接口调用是否 200 |
| `encrypted` | cookie 用本地用户密码派生 key 加密后落库 |

### 3.2.X 定时增量发现（登录后）

| ID | 需求 | 说明 |
| --- | --- | --- |
| F2.X.1 | 间隔可配 | 本地账号登录成功后，设置页可启用"定时增量发现"，间隔可选 30min / 1h / 6h / 12h / 24h。 |
| F2.X.2 | 触发条件 | 后端到点调度增量归档任务（见 F3.2）。 |
| F2.X.3 | 路径优先级 | 优先使用 A 的 cookie → 失败回退 B → 仍失败时等待 C 的下次推送。 |
| F2.X.4 | 通知 | 发现新内容时通过 WebSocket 给前端推一条 toast；不打扰，不弹窗。 |

### 3.3 归档内容范围

需要归档的内容类型与对应抖音接口已在 `packages/douyin-portal/src/api/urls.ts` 中实现拉取：

| 类型 | 数据源接口 | 归档物 |
| --- | --- | --- |
| **个人作品** | `user_post` | 视频文件 + 封面 + 元数据 |
| **喜欢的视频** | `user_like` | 视频文件 + 封面 + 元数据 |
| **收藏的视频** | `user_collect_video` | 视频文件 + 封面 + 元数据 |
| **收藏的音乐** | `user_collect_music` + `music_detail` | 音乐文件 (mp3) + 封面 + 元数据 |
| **收藏的合集** | `user_collect_mix` + `mix_detail` + `user_mix_detail` | 合集下所有视频 + 合集元数据 |
| **个人合集** | `user_mix` + `user_mix_detail` | 同上 |
| **稍后再看** | `watch_later_list` | 视频文件 + 封面 + 元数据 |
| **收藏夹（自建文件夹）** | `user_collect_folder` + `user_collect_folder_detail` | 同收藏的视频，但保留文件夹层级 |

**v1 明确不归档**：观看历史 (`user_record_*`)、关注 / 粉丝列表、评论、私信。

### 3.4 归档触发与状态

| ID | 需求 | 说明 |
| --- | --- | --- |
| F3.1 | 全量归档 | 一键触发：依次拉取上述 8 类内容的完整列表，去重后入队下载。 |
| F3.2 | 增量归档 | 仅拉取列表第一页，遇到已归档 ID 即停（按时间倒序，天然短路）。 |
| F3.3 | 单项归档 | 在浏览页对某一条视频 / 音乐 / 合集点击"归档"按钮，立即下载。 |
| F3.4 | 任务队列 | 后端维护下载队列，并发数可配置（默认 3），失败自动重试 3 次。 |
| F3.5 | 进度可见 | 前端 SSE / WebSocket 推送：当前队列长度、正在下载项、失败项、整体进度。 |
| F3.6 | 断点续传 | 视频下载支持 HTTP Range，中断后下次继续。 |
| F3.7 | 限速 | 可配置全局并发数 + 单连接限速，降低被风控概率。 |
| F3.8 | 失败可重试 | 失败列表页可手动重试 / 跳过。 |

### 3.4.X 抓取与下载器实现细则（借鉴 jiji262/douyin-downloader）

为避免重复造轮子，抓取 / 下载层的具体实现思路对照 [jiji262/douyin-downloader](https://github.com/jiji262/douyin-downloader)（Python，7.7k★，MIT）的同名模块**借鉴**而非整体引入。整体技术栈仍是本仓库自有的 Node + Fastify + Prisma + CloakBrowser，下表列出 v1 必须落地的 16 项能力及其参考位置。

| ID | 能力 | jiji262 参考位置 | 备注 |
| --- | --- | --- | --- |
| F3.X.1 | 登录与 cookie 持久化（A/B/C 三路统一抽象） | `auth/cookie_manager.py`、`auth/ms_token_manager.py` | 已在 §3.2 描述，本表强调结构对照 |
| F3.X.2 | 单视频下载（`/video/{aweme_id}`） | `core/video_downloader.py`、`core/api_client.py` | aweme_detail → 直采 video_url + 封面 + meta.json |
| F3.X.3 | 单图集 / note 下载（`/note/{id}`、`/gallery/{id}`） | `core/url_parser.py`、`core/video_downloader.py` 图集分支 | 多图打包成同目录 |
| F3.X.4 | 单合集下载（`/collection/{mix_id}`、`/mix/{mix_id}`） | `core/mix_downloader.py` | 展开合集所有 aweme |
| F3.X.5 | 单音乐下载（`/music/{music_id}`） | `core/music_downloader.py` | 优先直采音轨；无音轨则回落到关联 aweme |
| F3.X.6 | 短链解析（`v.douyin.com`、`v.iesdouyin.com`、bare host） | `core/url_parser.py` | 用户粘贴任何抖音分享链都能识别 |
| F3.X.7 | 用户主页批量（`/user/{sec_uid}` + mode: post/like/mix/music） | `core/user_downloader.py`、`core/user_modes/` | 翻页直到 has_more=0 |
| F3.X.8 | 当前账号收藏夹（`/user/self?showTab=favorite_collection`、mode: collect/collectmix） | `core/user_modes/` 收藏夹分支 | 受 §3.2 抖音身份层约束 |
| F3.X.9 | 附加资源（封面 / 原声 / 头像 / 元数据 JSON） | `core/metadata.py`、`storage/metadata_handler.py` | 与 §3.6 文件布局对齐 |
| F3.X.10 | 并发下载（默认 5，可在设置覆盖） | `core/downloader_base.py` 并发控制 | 与 §3.7 设置项保持一致 |
| F3.X.11 | 失败重试 + 指数退避（1s / 2s / 5s） | `core/retry_executor.py` | 配合 F3.4 三次重试上限 |
| F3.X.12 | 速率限制（默认 2 req/s） | `core/api_client.py` 限流 | 列表接口默认；下载层用单连接限速 |
| F3.X.13 | DB 去重 + 文件双重去重 | `storage/database.py`、`storage/file_manager.py` | DB 记 aweme_id 唯一约束；同 aweme 跨分类用硬链接 |
| F3.X.14 | 时间过滤（start_time / end_time） | `core/user_downloader.py` | 全量 / 增量任务都可裁剪 |
| F3.X.15 | 浏览器兜底（验证码 / 风控时切到 Cloak 直采） | `core/discovery.py` fallback 分支 | API 失败 → 走方案 A 同一会话 |
| F3.X.16 | 下载完整性校验（Content-Length 比对 + 失败自动重下） | `core/downloader_base.py`、`storage/file_manager.py` | 落地到 `.part` 文件，校验通过再 rename |

**进度可见性补充**：进度条 / 任务列表已在 F3.5 覆盖（前端 WS 推送），实现可参考 `utils/logger.py` 与 `progress.quiet_logs` 配置。

**签名策略备选**：若 CloakBrowser 「在登录页 `page.evaluate(fetch)` 让抖音 JS 自动加签」在 M2 真跑起来后出现内存 / 并发瓶颈，再考虑移植 `utils/abogus.py` + `utils/xbogus.py` 到 TypeScript 实现纯 undici 直请求。当前不动。

**明确不借鉴**（即使 jiji262 已实现）：评论归档、直播录制、热搜 / 关键词搜索、视频转写（Whisper / OpenAI Transcriptions）、推送通知（Bark / Telegram / Webhook）。

### 3.6 文件组织与存储

数据根目录默认：`~/.dolike-archive/` （可在设置中改）。

```
~/.dolike-archive/
├── db/
│   └── app.sqlite                          # 本地账号 + 元数据索引
├── accounts/
│   └── <douyin_sec_uid>/                   # 每个抖音账号独立目录
│       ├── posts/                          # 个人作品
│       │   └── 2024/2024-08/
│       │       └── <aweme_id>/
│       │           ├── video.mp4
│       │           ├── cover.jpg
│       │           └── meta.json
│       ├── likes/                          # 喜欢的视频（按归档时间分桶）
│       │   └── ...
│       ├── favorites/
│       │   └── default/                    # 默认收藏夹
│       │       └── ...
│       │   └── <folder_id>/                # 自建收藏夹
│       ├── watch_later/
│       ├── music/
│       │   └── <music_id>/
│       │       ├── audio.mp3
│       │       ├── cover.jpg
│       │       └── meta.json
│       └── mixes/
│           ├── created/<mix_id>/...        # 我创建的合集
│           └── collected/<mix_id>/...      # 我收藏的合集
└── tmp/                                    # 下载中断片
```

**规则**：

- 视频文件按**发布年-月**分桶（F3 需求"按发布时间保存视频并复现"）。
- `meta.json` 保留抖音原始接口返回的完整 JSON，避免后续字段需求要重抓。
- 同一 `aweme_id` 出现在多个分类（如既在作品又在喜欢）时，使用硬链接，不重复占磁盘。

## 4. 非功能需求

| 维度 | 要求 |
| --- | --- |
| 平台 | macOS / Linux / Windows 通用（Node.js + 浏览器）。v1 优先验证 macOS。 |
| 性能 | 单视频列表页 ≤ 500ms 渲染；本地全文搜索 1 万条 ≤ 200ms。 |
| 稳定性 | 下载失败可恢复；进程崩溃后任务队列可从 DB 恢复。 |
| 反风控 | 抓取走单线程 + 抖音原 cookie + 抖音真实 UA，并发由"下载"而非"列表抓取"环节承担。 |
| 安全 | 本地账号密码 bcrypt；抖音 cookie 加密；HTTP 服务仅绑定 127.0.0.1。 |
| 法务 | 仅归档当前登录账号自身可见的内容；README 明确"仅供个人备份，不得二次分发"。 |

## 5. 技术架构（高层，详见后端架构文档）

```
┌──────────────────────────────────────────────────────────────┐
│  Browser (Vue 3 + xgplayer)  ← 现有 douyin-portal 改造        │
│       │                                                       │
│       │  HTTP / WebSocket (localhost:7777)                    │
│       ▼                                                       │
│  Local Backend (Node.js / NestJS or Fastify)                  │
│   ├── Auth Module        本地账号 + session                   │
│   ├── Douyin Client      封装 cookie + 接口调用 + 风控应对     │
│   ├── Archive Service    列表拉取 + 入库 + 去重                │
│   ├── Download Worker    队列 + 限速 + 断点续传                │
│   ├── Library Service    本地浏览 / 搜索 / 筛选                │
│   └── Media Server       静态文件 + Range 流式分发             │
│       │                                                       │
│       ▼                                                       │
│  SQLite (元数据) + 文件系统 (视频/音频/封面)                   │
└──────────────────────────────────────────────────────────────┘
```

外部依赖：

- 抖音网页端接口（既有 url 表已覆盖大部分需求）。
- 不依赖任何抖音官方开放平台 OAuth（个人账号无法申请）。

## 7. 风险与开放问题

| 风险 | 缓解 |
| --- | --- |
| 抖音风控升级 / 接口签名变化 | 抓取层抽象出 `DouyinClient`，单独维护签名生成（参考既有 `mafqla/douyin-api` 项目） |
| Cookie 频繁失效 | 失败检测 + UI 提示重新导入；v2 引入 Playwright 自动续期 |
| 视频 url 过期（抖音 url 通常 6-12h 时效） | 下载流水线优先级高于列表入库；列表入库后立即触发下载 |
| 法务争议 | 限定"仅本人可见内容 + 本机存储 + 不分发"，README 加免责声明 |
